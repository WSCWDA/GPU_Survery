\section{Introduction}

GPU-centric computing now dominates both large-scale AI training and modern HPC simulation, while data footprints have expanded from dataset-scale ingest to frequent model-state persistence at cluster scale. As a result, storage and I/O orchestration increasingly determine end-to-end performance, even in systems with abundant accelerator FLOPS. Evidence from ML input pipelines and distributed caching systems shows that data staging, shuffling, and locality management can directly cap GPU utilization when storage services are not co-designed with compute phases \cite{murray2021tfdata,hoard2018,zhang2025hpcstoragesurvey,rajbhandari2021zeroinfinity}.

The fundamental mismatch is architectural: mainstream storage software remains CPU-centric. In conventional paths, the host CPU coordinates submission, metadata, synchronization, and completion; data frequently traverses host-managed stages before becoming GPU-consumable. Hence, \emph{the CPU remains on the critical path of I/O}, and host-side control overhead scales poorly under massively parallel GPU workloads \cite{linux_io_uring,spdk_overview,nvme_spec,silberstein2013gpufs,shahar2016gpufs}.

Recent technologies (e.g., RDMA, GPUDirect RDMA, GPUDirect Storage) improve transfer efficiency by reducing copies and bypassing host bounce buffers. However, these mechanisms predominantly optimize \emph{how bytes move}, not \emph{how storage control evolves}. Consequently, many systems retain host-dominant control and metadata paths, which limits end-to-end gains in complex training/serving pipelines \cite{infiniband_spec,nvidia_gdrdma,nvidia_gds_blog,nvidia_gds_overview,traub2023compstoragesurvey}.

This survey adopts a broader systems perspective: the key transition is from GPUs as passive consumers to GPUs as active participants in storage interaction and, increasingly, storage control. We use \emph{GPU-integrated storage} to describe architectures that co-design control logic, data movement, and deployment assumptions around accelerator execution contexts. Representative systems from GPUfs to BaM and GeminiFS illustrate this progression from CPU orchestration toward GPU-participatory operation \cite{silberstein2013gpufs,qureshi2023bam,geminifs2025}.

Accordingly, our scope is not limited to transport optimization. We analyze the full narrative---\textbf{CPU-orchestrated $\rightarrow$ GPU-aware $\rightarrow$ GPU-direct $\rightarrow$ GPU-driven}---across control plane, data path, and deployment model, and connect these dimensions to workload-driven design constraints in AI/HPC environments \cite{nvmeof_spec,pcie_spec,liu2017rackscale,murray2021tfdata,zhong2024distserve}.

Our contributions are fourfold:
\begin{enumerate}
    \item \textbf{Systematic co-evolution view.} We synthesize the architectural transition from CPU-orchestrated storage stacks to GPU-integrated and emerging GPU-driven designs \cite{nvidia_gds_blog,silberstein2013gpufs,qureshi2023bam}.
    \item \textbf{Unified taxonomy.} We define a three-dimensional taxonomy spanning control plane, data path, and deployment model.
    \item \textbf{Design-dimension synthesis.} We distill cross-system principles for layout, scheduling, memory hierarchy, and resilience under GPU-intensive workloads \cite{geminifs2025,rajbhandari2021zeroinfinity,nvidia_gds_overview}.
    \item \textbf{Problem-driven research agenda.} We identify concrete open problems for GPU-native I/O control, disaggregated deployments, and unified programming abstractions \cite{kwon2023vllm,zhong2024distserve,zhang2025hpcstoragesurvey}.
\end{enumerate}
